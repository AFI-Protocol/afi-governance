## 1. Verified evidence restated (input, not authority)

1. **The runtime resolves the full invocation identity and then discards it.** `ProviderRuntime.invoke` (`afi-reactor/src/providers/providerRuntime.ts:54`) resolves ProviderInstance (id @ recordVersion), Provider, adapter (adapterId @ adapterVersion via the static fail-closed registry, `:108`), governed model, tenant, and credential scope — and returns only the bare category result: `CategoryResult = {category, ...payload}`. No identity, hash, or status fact survives the call. The executor already returns a per-node runtime record array (`executor.ts:608-624`) that `graphScoringService.ts` discards. Invocation provenance can therefore be captured **in the same pass, with no provider re-call** — the capture path exists and is dormant.
2. **Per-lane hashes are computed nowhere today.** The evidence-layer hash law's tag table carries a dormant `laneOutput: "afi.d2.lane-output"` entry that no code uses (`canonicalHashV1.ts:88`); `EnrichmentProvenance.laneOutputHash` (the PR-O1 pointer) is a dormant draft field; the only lane-content commitment is the composition-law `enrichmentHash` over the merged bundle (`graphScoringService.ts:165-192`).
3. **Two distinct canonicalization laws both stamp `afi.hash.v1`.** Evidence law: preimage `"afi.hash.v1" + "\n" + domainTag + "\n" + canonicalJson`, floats rejected except declared decimal-projection keys (`afi-reactor/src/evidence/provenance/canonicalHashV1.ts`, `hashProjection.ts`). Composition law: `canonical-json-hashing.v1` (RFC-8785-aligned), domain tag carried but not hashed, shortest-round-trip floats admissible (`afi-reactor/src/pipeline/hashing.ts`). `inputHash`/`outputHash` use the former; all composition hashes (including `enrichmentHash`) use the latter. A third law exists at the Tiny Brains boundary: Python canonical JSON (`sort_keys`, compact separators) over request/response payloads (`afi-tiny-brains/tiny_brains_service/orchestration/invocation_record.py:25-35`).
4. **Tiny Brains already produces the aiMl invocation facts, but they die in logs.** `tiny-brains.aiml-invocation.v1` (profile/resolver/expert identities and versions, postures, statuses, per-expert output hashes, artifact fingerprints, input/output hashes, code/config fingerprint — hashes and identifiers only, no raw payloads, prompts, paths, or secrets) is built for every invocation, including failures, and emitted only as a log line (`service.py:120-126`); the HTTP response carries only a profile echo (`models.py:77-89`), and the reactor client validates `profileId` but not the echoed `profileVersion`. No provider/instance/credential record carries any content fingerprint today — record identity is (id, recordVersion).

Nothing is deployed, published, or externally consumed.

The governed pipeline contract already provides the fail-fast vocabulary (`critical` defaults to true; `degrade` is structurally allowed only when `critical` is explicitly false — `afi-config/schemas/pipeline/v1/pipeline.schema.json:139-168`).

V3 is **v2's record shape carried forward unchanged** — the same twelve required properties (schema const excepted), the same reused `afi.scored-signal.v1` / `afi.provenance-record.v1` / uwrProfile-stamp / `afi.composition-ref.v1` sub-shapes, the same optional `recordVersion`/`supersedesRecordHash`, the same lifecycle/finality binder, `additionalProperties:false` at every level — **plus exactly three new REQUIRED properties**:

1. **`providerInvocations`** — the five-lane provider invocation proof collection (D-EV3-2);
2. **`recordHash`** — the full-record integrity commitment (D-EV3-4);
3. **`replayHash`** — the deterministic semantic/replay commitment (D-EV3-4).

V3 is the **active write contract and the only current evidence contract**: no dual V2/V3 mode, no feature-flagged V2, no fallback projection, no compatibility alias. The freeze D-PBF-10 and D-FLPR-5(3) placed on Evidence V2 dissolves into this decision: V2 ceases to be an active contract, and the Evidence-V2 freeze guards (including the reactor CI freeze-guard test) are superseded by V3-shaped guards rather than deleted silently. Both vendored governed-schema closures (afi-reactor, afi-infra) **gain the V3 and proof-schema members** and are re-pinned to the post-merge afi-config commit. Signal identity is untouched: `signalId` semantics, USS v1.1, the strategy triple, `afi.scored-signal.v1`, and identifier-continuity law are consumed exactly as OBJ-GOV/LIFE-GOV fixed them.

**Scope-guard.** Fixes the **V3 record identity and sole-current status** only. It redesigns no reused sub-shape (`afi.scored-signal.v1`, `afi.provenance-record.v1`, `afi.composition-ref.v1`, the uwrProfile stamp are shape-untouched), renames no field of the carried-forward v2 core, changes no lifecycle vocabulary, and extends no lifecycle implementation beyond `SCORED`.

## 3. D-EV3-2 — the per-lane provider invocation proof

**Decision.** A new closed contract **`afi.provider-invocation-proof.v1`** (afi-config family, D-FCP-3) records one successful provider invocation per category. On every V3 record, `providerInvocations` MUST contain **exactly five** proofs — one per governed category `technical`, `pattern`, `sentiment`, `news`, `aiMl` (D-FCP-1 namespace, casing exact) — **unique by category and deterministically ordered by ascending case-sensitive lexicographic category name (`aiMl`, `news`, `pattern`, `sentiment`, `technical`)**. Each proof binds, with non-secret facts only:

1. **Category-result identity** — category; the governed result schema id, enumerated exactly (`afi.enrichment.technical.v1`, `afi.enrichment.pattern.v1`, `afi.enrichment.sentiment.v1`, `afi.enrichment.news.v1`, `afi.enrichment.aiml.v1` — contract ids are lowercase; only the runtime category marker is camelCase `aiMl`); the category-result hash (the exact result that entered the join); result status `succeeded` (a scored V3 record admits no other status — under D-EV3-5 a failed lane yields no scored evaluation at all).
2. **Provider identity** — providerId, provider recordVersion, provider record fingerprint (D-EV3-4), executionClass, deterministic/probabilistic posture.
3. **ProviderInstance identity** — providerInstanceId, instance recordVersion, instance record fingerprint (the non-secret configuration commitment), and the governed `model` identifier where the instance declares one.
4. **Adapter identity** — adapterId, adapterVersion, transport kind (`in-process` | `http`).
5. **Credential binding** — exactly one of: explicit keyless posture (`mode: "keyless"`), or an opaque CredentialRef binding (`mode: "credentialRef"` with credentialKind, the opaque credentialRef id, its recordVersion, and its governed status) — never a secret, secret-derived hash, header, token, or credentialed URL (D-EV3-6).
6. **Invocation input and output** — the normalized invocation-input hash and the provider-result hash (D-EV3-4); and, as the only per-lane source-reference field admitted in this version, the technical lane's non-secret price-source identifier (**no other lane declares a source-reference field in `afi.provider-invocation-proof.v1`**).
7. **aiMl nested proof** — required exactly when category is `aiMl`, structurally forbidden otherwise (D-EV3-3).

Proof facts are **carried, never consumed**: no category node, join, analyst, scorer, UWR, or decay path may read a provider invocation proof; lane vendor-neutrality at the enrichment seam (D-FLPR-1 scope-guard) is unchanged. The proof collection is provenance output, not runtime input.

**Scope-guard.** Fixes the **proof contract, the exactly-five/unique/ordered law, and the carried-not-consumed rule** only. It adds no provider, adapter, instance, or credential record; changes no adapter normalization or governed enrichment payload shape; admits no per-lane field beyond those enumerated here; and creates no operational-diagnostics store.

## 4. D-EV3-3 — the aiMl nested invocation proof and the bounded Tiny Brains boundary amendment

**Decision.** The `aiMl` proof MUST additionally carry **`afi.aiml-invocation-proof.v1`**: the deterministic, non-secret projection of the successful `tiny-brains.aiml-invocation.v1` record — orchestration profileId and profileVersion; resolverId and resolverVersion; the service code/config fingerprint; the normalized Tiny Brains input hash and final output hash; invocation status `succeeded`; and, per required expert: expertId, expertVersion, deterministic/probabilistic posture, status `succeeded`, expert output hash, and the model artifact fingerprint set. **Volatile timing facts (startedAt/endedAt/durations) are structurally excluded.** The owner explicitly authorizes these service-internal identities (expert ids, artifact fingerprints) to appear in canonical evidence: they are non-secret provenance facts of the invocation that occurred. Vendor-neutrality of the enrichment seam is preserved by the carried-not-consumed rule (D-EV3-2): the reactor records these facts as opaque provenance material and no lane, join, or scoring logic may branch on them; D-PBF-3's subordination posture (one resolved `aiMl` result, no scoring authority) is unchanged.

**One bounded additive Tiny Brains amendment is authorized** so these facts can cross the service boundary: the successful `POST /predict/froggy` response gains one closed, non-secret `invocation` block carrying exactly the projection above (hashes and identifiers only; no raw candles, prompts, paths, weights, or expert payloads). The existing `outputHash` law is unchanged — it commits to the prediction payload exclusive of the `invocation` block. The reactor client's profile-echo law is hardened to validate `profileVersion` as well as `profileId`, fail-closed. **Nothing else in Tiny Brains changes**: no orchestration, profile, resolver, expert, artifact, or kernel change; failure responses remain bounded operational diagnostics that never yield a scored V3 record.

**Scope-guard.** Fixes the **nested-proof contract and the single additive response block** only. It grants Tiny Brains no authority, changes no `afi.enrichment.aiml.v1` payload shape, adds no second aiMl result, and authorizes no other Tiny Brains API change.

## 5. D-EV3-4 — hash law: domain tags, projections, PR-O1 resolution, and the replay separation

**Decision.**

1. **Six new canonical-hash domain tags are registered**, narrowing CH-O2 exactly as D-FCP-7 did: `afi.d2.provider-record`, `afi.d2.provider-instance-record`, `afi.d2.provider-invocation-input`, `afi.d2.provider-result`, `afi.d2.evidence-record`, `afi.d2.evidence-replay`. **The dormant `afi.d2.lane-output` reservation is re-homed**: its unused entry in the evidence-layer law's tag table (`canonicalHashV1.ts` `D2_DOMAIN_TAGS.laneOutput`) is removed under D-EV3-8, and the tag is activated for the composition-law `categoryResultHash` — six new tags plus one re-homed reservation, seven in active use.
2. **Every new hash this decision creates is computed under the existing composition canonicalization law** (`canonical-json-hashing.v1`, the law already committing lane content via `enrichmentHash`) and carried as a `CanonicalHash v1` object. The evidence-layer law and its uses are untouched: `inputHash` (`afi.d2.signal-input`) and `outputHash` (`afi.d2.scored-output`) remain byte-stable, as do `pluginSetHash` and the per-artifact composition-law rules. (`manifestHash` and `analystConfigHash` values move only through the governed D-EV3-5 manifest amendment, not through any law change.)
3. **Per-lane hashes**: `invocationInputHash` commits to the normalized, non-secret invocation input projection (the adapter's resolved input facts; credential material, headers, and volatile fields structurally excluded by the projection's closed definition); `providerResultHash` commits to **the validated category result minus its `category` property** (the provider's normalized payload); `categoryResultHash` (tag `afi.d2.lane-output`) commits to the full category result consumed by the join.
4. **PR-O1 is resolved**: per-lane commitments live as the five `categoryResultHash` values inside the invocation proofs, **and** the single bundle `enrichmentHash` is retained at the composition layer. `afi.provenance-record.v1` keeps its shape (its optional single `enrichmentHash` stands); its `x-afiOpenItems.PR-O1` note is updated — a metadata annotation, not a shape change — to record this resolution, and the dormant draft `EnrichmentProvenance` object it referenced is subsumed by the invocation proof and deleted under D-EV3-8.
5. **Tiny Brains hashes are opaque commitments, not CanonicalHash objects**: they are recorded as plain digest fields explicitly pinned to the service's own canonicalization law (`tiny-brains.hash.v1` — Python canonical JSON: lexicographically sorted keys, compact separators, UTF-8, Python shortest-round-trip float repr). The reactor client MUST verify the service's final `outputHash` by exact recomputation over the received prediction payload under a KAT-proven implementation of that law, fail-closed. **The verification law's float domain is pinned**: the closed response contract carries exactly one float field, `convictionScore`, declared 6-decimal-quantized and clipped to [0, 1] by the resolver law; the recomputation implementation MUST reproduce Python float repr over that domain exactly (including the integral forms `0.0`/`1.0` and sub-`1e-4` exponent forms), MUST be proven by Python-generated known-answer vectors covering the boundary and quantization cases, and MUST fail closed on any float outside its proven domain. The aiMl cross-check chain is thereby closed end-to-end: Tiny Brains output hash ↔ verified wire payload → deterministic normalization → `providerResultHash` → `categoryResultHash` → the aiMl proof in V3.
6. **Record-level hashes**: `recordHash` (tag `afi.d2.evidence-record`) commits to the full V3 record minus {recordHash, replayHash}; `replayHash` (tag `afi.d2.evidence-replay`) commits to the **replay projection** — the record minus {recordHash, replayHash, lifecycleState, finalized, recordVersion, supersedesRecordHash}. This also gives the pre-existing `supersedesRecordHash` link its defined computation for V3 records.
7. **The canonical/replay separation is explicit**: (a) **immutable historical invocation facts** = the identity and hash facts inside the proofs; (b) **the deterministic semantic/replay projection** = the replayHash preimage — two evaluations of the same canonical inputs through the same composition MUST produce identical replay projections and identical `replayHash` values; (c) **non-canonical operational diagnostics** = wall-clock timings, attempt counts, retry/rate-limit facts, and transport noise, which MUST NOT appear anywhere in the canonical V3 record (they remain runtime logs). **No volatile or operational timestamp or duration may enter any hashed preimage of this decision**; domain-declared normalized evidence timestamps remain admissible exactly as the District 2 hash doctrine already provides (e.g. the `afi.scored-signal.v1` optional `evaluatedAt`, which is admissible only as deterministic domain material, never wall-clock).
8. **Proof obligations**: governed known-answer vectors for every new projection; mutation tests proving each load-bearing field moves exactly the correct hashes and trips the correct cross-check; **structural exclusion** (closed, enumerated preimage projections) plus synthetic-marker tests asserting no tested surface admits a secret or diagnostic marker.

**Scope-guard.** Fixes the **tag registrations and re-homing, law assignments, PR-O1 resolution, boundary-verification law, and projection separation** only. It amends neither canonicalization law's algorithm, no existing KAT, no existing hash field's preimage, and — beyond removing the one dormant `laneOutput` table entry — no existing domain-tag use; `afi.hash.v1` remains the stamp of both existing laws exactly as before.

## 6. D-EV3-5 — evaluation completeness, runtime capture, District Two construction, and fail-closed mismatch law

**Decision.**

1. **A scored evaluation requires all five lanes to succeed.** The five-lane degrade allowance is retired by **one bounded pipeline-manifest failure-policy amendment**: a successor manifest **`froggy-trend-pullback v1.3.0`** identical to v1.2.0 except that the five category lane nodes drop `critical:false` / `failurePolicy:"degrade"` (becoming fail-fast under the governed default), registered as a new registry file under the pipelines change-control rule (files are never edited in place), with the analyst-strategy registration re-pinned to it. **Graph geometry (nodes, edges, join), plugin bindings, and every `providerInstanceRef` are byte-unchanged**; `manifestHash` and `analystConfigHash` move exactly as the governed composition-hash-movement mechanism provides (the D-FLPR-5(5) precedent), `pluginSetHash` stays byte-identical, and regenerated composition-bearing goldens must document the field-level intentional diffs with all scored-value fields byte-unchanged. A failed or degraded lane now yields **no scored evaluation, no scored signal, and no evidence record** — bounded operational diagnostics only. This aligns runtime law with evidence law: the exactly-five-successful proof requirement (D-EV3-2) is satisfiable by construction on every scored run.
2. **Invocation proof facts are captured inside the one live graph pass** at the provider runtime seam — where the identity chain is already resolved (§1.1) — attached to the run's execution state, and carried to District Two with the scored result. **Evidence describes the invocation that occurred; it never re-calls a provider**: no evidence builder may invoke an adapter, re-fetch market data, re-run Tiny Brains, recreate a category result, or accept invocation facts detached from the current run.
3. Exactly **one** current Evidence V3 builder exists in District Two; it receives the completed scored evaluation state and the five bound proofs, and it MUST fail closed on: a missing, duplicate, or unknown category proof; mis-ordered proofs; a category/result-schema mismatch; a ProviderInstance or adapter identity that differs from the boot-verified governed registry resolution; a category-result hash that does not equal the recomputed hash of the result the analyst path actually consumed; a Tiny Brains profile/resolver/expert projection that fails its cross-checks; a credential-binding fact inconsistent with the instance's governed record; a composition, analyst, scorer, UWR, or decay identity mismatch; a schema-validation failure; or a canonical-hash mismatch. A failed lane, a failed cross-check, or a failed validation yields **no scored V3 record** — never a fabricated proof, never a downgraded record, never a V2 write.
4. **Provenance capture requires no adapter or instance re-versioning**: the governed enrichment payloads, adapter normalization contracts, transport targets, credential law, and instance selection are all unchanged, so the five adapters and ProviderInstances retain their current identities and versions. **The owner expressly determines** that the D-EV3-3 boundary-verification hardening (the `invocation`-block requirement and the `profileVersion` echo check in the aiMl transport client) is within the identity of adapter `afi-adapter-aiml-tiny-brains@1.1.0`: it strengthens transport-response verification of newly available provenance facts and changes no normalized output, no transport target, and no credential behavior. (Any future change to a normalization contract, transport target, or credential law still re-versions under existing PBF/FLPR law.)

**Scope-guard.** Fixes the **all-five completeness amendment, the capture seam, the sole-builder rule, the fail-closed mismatch law, and the no-re-versioning determination** only. It changes no graph geometry, no provider selection, no join semantics for complete runs, and no scored value on complete evaluations; it creates no second executor, no evidence orchestration engine, no operational provenance store, and no failed-invocation evidence artifact.

## 7. D-EV3-6 — credential-safety law

**Decision.** The BYOK boundary (D-PBF-7) extends to evidence with enumerated negative proofs. No V3 record, proof, hash preimage, fixture, golden, log line, error, or snapshot may contain: a secret value; an API key; a bearer token; an authorization header; a secret-manager payload or path; a credential-bearing URL; a hash or fingerprint derived from secret material; a raw provider response body; a raw Tiny Brains candle series; a private model filesystem path; an unbounded provider error string; or another tenant's credential identity.
